# 🎯 AI 编程全局规范 (Global Rules)

> 语言无关、模块化、文档驱动的 AI 协作规范

---

## ⚠️ 语言要求

**所有 AI 回复、代码注释、文档必须使用中文**

---

## 🚨 一、核心铁律

### 1.1 入口文件只做路由
```
❌ 禁止在入口文件写业务逻辑
✅ 入口文件只导入和调用模块
```

### 1.2 功能必须模块化
```
一个功能 = 一个独立目录
一个目录 = 必须有 README.md
功能多了 = 创建更多模块目录（不是让单文件变大）
```

### 1.3 逻辑必须分离
```
UI 代码 → components/ 或 views/
业务逻辑 → services/ 或 hooks/
数据处理 → utils/ 或 helpers/
```

---

## 📁 二、项目结构

```
project-root/
├── RULES.md           # 开发规范
├── README.md          # 项目说明
├── docs/              # 文档目录
│   └── modules/       # 模块说明
├── src/
│   ├── modules/       # 功能模块
│   │   ├── feature-a/
│   │   │   ├── README.md
│   │   │   └── ...
│   │   └── feature-b/
│   └── shared/        # 共享资源
```

> 📌 所有目录和文件名使用英文 + 小写 + 连字符
> *(注：由于 Python 的 import 限制，内部 Python 文件仍保留下划线 `_` 格式，以确保代码可用性，但模块和顶层目录严格遵循小写连字符)*

---

## 📝 三、模块文档 (必须)

每个模块目录下必须有 `README.md`：

```markdown
# Module Name

## 职责
一句话说明

## 文件
| File | Purpose |
|------|---------|

## 依赖
- dependency list

## 使用
code example
```

---

## 🧩 四、模块化原则

### 4.1 单一职责
- 每个模块只做一件事
- 功能多了 → 创建更多模块

### 4.2 拆分原则（非硬性限制）
```
单个文件超过 300 行 → 考虑拆分
单个函数超过 50 行 → 考虑拆分
逻辑复杂 → 提取到单独模块
```

> ⚠️ 这是建议值，不是硬性限制。核心原则是：**功能多就多建模块，不是让文件变大**

### 4.3 正确做法
```
❌ 一个 1000 行的大文件
✅ 5 个 200 行的小模块
```

---

## 🤖 五、AI 协作规范

### 5.1 AI 工作流程
```
1. 先读 RULES.md
2. 再读目标模块 README.md
3. 理解后再编码
```

### 5.2 代码修改原则
```
- 不随意删除现有代码
- 不擅自修改无关文件
- 修改前先说明意图
- 保持代码风格一致
```

### 5.3 新增功能流程
```
1. 创建模块目录 → modules/new-feature/
2. 写 README.md → 说明职责
3. 写代码 → 在模块内部
4. 入口调用 → 只加一行
```

## 🔐 六、安全规范

```
❌ 硬编码密钥       → ✅ 环境变量
❌ URL 带密钥       → ✅ HTTP Header
❌ 打印敏感信息     → ✅ 日志脱敏
```

---

## 📦 七、命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 目录 | 小写+连字符 | `user-auth` |
| 文件 | 小写+连字符 | `image-utils.ts` |
| 组件 | 大驼峰 | `UserProfile` |
| 函数 | 小驼峰 | `getUserInfo` |

---

## 💡 核心原则

1. **模块化** — 功能多就多建模块目录
2. **文档化** — 每个模块有 README
3. **分小块** — 大文件拆成小文件
4. **分离** — UI、逻辑、数据分开
5. **中文** — 所有沟通使用中文
6. **英文** — 所有目录/文件名用英文

---

> 📌 此规范适用于任何编程语言和框架
